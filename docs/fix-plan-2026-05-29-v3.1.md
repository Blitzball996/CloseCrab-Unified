## 5\. rm/危险命令拦截

### 关于 MSYS_NO_PATHCONV(v3.1 修正 v3 的理由)

**v3 撤回 MSYS_NO_PATHCONV,结论可保留,但 v3 给的理由是错的。**

*   v3 说"设了会破坏 `bash -c \"ls /usr\"`"——**不成立**:`ls` 是 MSYS 自带程序,`/usr` 由 MSYS 内部挂载表解析,与参数路径转换无关,设不设都不受影响。
*   事实上 `findstr` 是 Windows 原生命令,**`MSYS_NO_PATHCONV=1` 恰好能修**那个 `/i→I:/`、`/n→N:/` 的 mangling。
*   它真正的副作用是:这是个**全局开关**,会让"故意想把 POSIX 路径自动转成 Windows 路径再喂给原生 .exe"的场景失效。

**v3.1 的正确做法(优先级从高到低)**:

1.  **首选:压根别从 bash 调 Windows 原生命令。** 用 MSYS 自带 `grep -in lang /g/CMakePJ/.../index.html`(`G:\` 对应 `/g/`),或直接用 §4 引导的内建 Grep/Read 工具。这样从源头没有 mangling,也不碰全局开关。
2.  **次选:若某条命令确实必须调原生 exe,只在那一次 spawn 上加** `MSYS2_ARG_CONV_EXCL='*'`(或精确列出要排除的参数前缀),做**局部**关闭,而非全局 `MSYS_NO_PATHCONV`。
3.  全局 `MSYS_NO_PATHCONV=1` 不推荐,理由是上面的"全局副作用",**不是**因为它会破坏 `ls /usr`。
