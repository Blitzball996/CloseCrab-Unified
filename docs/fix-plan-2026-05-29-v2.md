## 🔥 新发现 1 — 缓存不命中的真实原因（覆盖 v1 P1 缓存部分）

**v1 说**："易变内容混进 cache_control 前缀。"

**实测代码**：
- `QueryEngine::buildSystemPrompt()`（QueryEngine.cpp:60-176）已经做了缓存，且 cwd/model/mode 是稳定的 → **system prefix 本身是稳定的**，不是 v1 猜的"时间戳/计数污染前缀"。
- 真正的破坏者是 **ToolSearch defer_loading**（QueryEngine.cpp:210-220）：tools 数组随 ToolSearch 调用动态增长，每轮 tools 都不同 → **`tools.back()["cache_control"]` 的 tools 前缀每轮都变** → cache 写了等于白写。
- 第二个破坏者是 **RemoteAPIClient.buildRequestBody 的 microcompact in-place 改写**（RemoteAPIClient.cpp:48-81）：当 messages > 200KB 时，**就地修改老 tool_result 的 content** → 消息历史前缀字节级变化 → 命中不了上一轮在同一位置写的 cache。
- 第三个：当前设了 3 个断点（`system` 末尾 + `tools` 末尾 + `last message` 末尾），且 **没有"静态前缀"断点**的明确分工。

**修改项（替换 v1 P1）**：

| 改动 | 文件:行 | 做法 |
|---|---|---|
| 冻结 tools 数组（前 N 轮发全量，禁用 defer_loading 增量） | `QueryEngine.cpp:210` | 复用 `ALWAYS_LOAD` 集合作为整个会话的固定 tools 列表；ToolSearch 发现的 tool 只在**本轮单独发**，下一轮回退到固定列表。或彻底关掉 defer_loading，按 JackProAi 那样所有 tool 一次发完（59 tools 顶天 20KB，比每轮丢 70KB cache 划算）。 |
| microcompact 改为"截断 + 不修改老消息字节" | `RemoteAPIClient.cpp:48-81` | 不要 in-place 改写老 tool_result。改成：发送时**只裁切 messages 数组的开头**（drop oldest N），保留剩下消息字节完全一致。或者用 Anthropic 官方 `context_management.edits` 字段（apiMicrocompact.ts 的 `clear_tool_uses_20250919` 策略），让服务端去清理。 |
| 双断点策略 | `RemoteAPIClient.cpp:35-100` | 断点 A：`tools.back()`（静态，**只在 tools 数组确定不变后才设**）；断点 B：`messages` 倒数第 2 条的末尾（rolling，给下一轮命中本轮写的 cache 用）。删掉 system 自带断点（system 已经 prefix 在 tools 之前，被断点 A 覆盖）。 |
| 加 cache 命中日志 | `RemoteAPIClient.cpp` 响应处理处 | 解析 `usage.cache_read_input_tokens` / `cache_creation_input_tokens` 打到 spdlog，**每个请求 1 行**，便于回归看命中率。 |
| 拿官方 API 直连验证 | 配置层 | 临时换 `https://api.anthropic.com` + 真 API key 跑一遍同一任务，对比缓存命中数据。若官方命中、yikoulian 不命中 → 中转不支持 prompt-cache，必须换中转或走官方。 |

**预期收益**：账单表显示 9 条请求烧 $7.38。命中后写量从 77K 降到首轮 77K + 后续 ~5K 增量、读量 77K × $0.85/1M × 1.7 ≈ **$0.07/请求**，总成本降到 ~$0.8。**省 8-10×。**

## 🔥 新发现 2 — Windows 路径根因在 CloseCrab 这边没做处理（覆盖 v1 P0 Windows 路径）

**v1 说**："不要把 Windows 原生命令丢给 Git Bash。"对，但**没说 CloseCrab 自己怎么改**。

**实测代码**：
- `BashTool.h:139` 直接 `"C:/Program Files/Git/bin/bash.exe" -c "<cmd>"`，**完全没设 `MSYS_NO_PATHCONV` 或 `MSYS2_ARG_CONV_EXCL`** → LLM 写出来的 `findstr /i /n "lang" "G:\..."` 进入 Git Bash 后 `/i` 被转成 `I:/`、`G:\...` 被吃掉反斜杠 → 就是 bug.txt 第 58-78 行看到的"FINDSTR: 无法打开 I:/"灾难。
- 同样没做 cwd 切换 — 当前没看到 `chdir(ctx.cwd)` 或 `CreateProcess` 的 `lpCurrentDirectory` 参数，所以 LLM 的 `cd ...` 也容易失效。
- JackProAi 的 `windowsPaths.ts` 提供了 `posixPathToWindowsPath` / `windowsPathToPosixPath` 双向转换 + `findGitBashPath()` 自动发现，CloseCrab 没有等价工具，硬编码 `"C:/Program Files/Git/bin/bash.exe"`（用户机器装在别处就废了）。

**修改项**：

| 改动 | 文件:行 | 做法 |
|---|---|---|
| 进程环境注入 `MSYS_NO_PATHCONV=1` 和 `MSYS2_ARG_CONV_EXCL=*` | `BashTool.h:124-200` (`executeWithTimeoutWin32`) | `CreateProcessA` 之前构造 env block，**强制关掉 MSYS 的 POSIX→Win32 路径转换**。这一行改动直接消除 `/flag → I:/`、`\path → 吃斜杠`两类故障。 |
| 设 cwd | `BashTool.h:150` | `CreateProcessA` 第 8 个参数 `lpCurrentDirectory` 传 `ctx.cwd.c_str()`，别让子进程继承我们的进程目录。 |
| Git Bash 路径自动发现 | 新文件 `utils/WindowsBashFinder.h` | 仿 JackProAi `findGitBashPath()`：先看环境变量 `CLAUDE_CODE_GIT_BASH_PATH`，再 `where git` → 推导 `bash.exe`，再回退到默认。 |
| 系统提示加一段"Windows 命令规则" | `QueryEngine.cpp:72` (Behavior Rules) | 在 system prompt 加：`On Windows under Git Bash: prefer POSIX paths (/g/CMakePJ/...) over G:\\...; do NOT call findstr/cmd builtins — use bash builtins (grep, ls); avoid inline node -e / sed -i / python -c — write a temp file and execute it.`这条不是给 CloseCrab 改代码，是**直接堵掉 LLM 跑那种灾难命令的源头**。 |
| `chcp 65001` + UTF-8 | `BashTool.h:139` | `CreateProcessA` 之前调一次 `SetConsoleOutputCP(CP_UTF8)`，再把 `LANG=zh_CN.UTF-8` 注入 env block。配合 `ensureUtf8(output)` 已经在做的转换，覆盖 mojibake。 |

## 🔥 新发现 3 — "写失败后删文件"是 LLM 决策行为，不是 CloseCrab 工具 bug（覆盖 v1 P0 数据安全）

**v1 说**："删掉'写失败 → 删文件重建'的逻辑。"

**实测代码**：
- `FileWriteTool.h:32-68` 只做"打开 → 写 → 失败返回 fail message"，**没有删文件的代码**；`FileEditTool.h` 同理。
- bug.txt 第 488-494 行看到的删文件流程：
  ```
  HTML 文件也被锁住了... → cp 备份失败 → rm "*.html" && echo "deleted"
  ```
  这是 LLM 看到 `UNKNOWN: open ...` 之后**自主决定**调 BashTool 跑 `rm`。CloseCrab 工具层没问题，问题在**没有任何护栏拦住 LLM 跑 `rm 用户文件`**。

**修改项（替换 v1 P0 数据安全部分）**：

| 改动 | 文件:行 | 做法 |
|---|---|---|
| BashTool 增加"危险命令"分类 | `BashTool.h:276` (`isSafeCommand`) 旁边新增 `isDangerousCommand` | 匹配 `rm `、`rm -`、`del `、`rmdir`、`> /` 等；命中时 `checkPermissions` 返回 `ASK_USER` **且** 在 prompt 里塞一段"about to delete files: X, Y, Z; confirm?"。 |
| 自动备份兜底 | 新工具 `BackupGuard`（也可以塞在 BashTool 拦截层） | `rm` 执行前把目标路径 copy 到 `.crab/backups/<时间戳>/`；失败则中止。 |
| 写失败错误信息加诱导 | `FileWriteTool.h:48-50` & `FileEditTool.h:73,107,167` | 失败信息改成：`"Cannot open file (likely locked by editor/Live Server). DO NOT delete the file. Ask the user to close the application, then retry."`，**主动反提示 LLM 不要走删除路径**。 |
| 系统提示加禁令 | `QueryEngine.cpp:72` (Behavior Rules) | 加一行：`NEVER use rm/del on user source files. If a write fails with UNKNOWN/EBUSY/locked, report to user and ASK before any cleanup. Files without backup must never be deleted to "rebuild".` |

## 🟡 v1 P1 循环控制 — 已经部分实现，但阈值/动作还差

**实测代码**（QueryEngine.cpp:486-1019）：
- `kRepeatBreakThreshold=7`, `kSameToolBreakThreshold=15`, **已经会强制 break**（line 996-1004），不是 v1 说的"只 inject warning"。
- 警告阈值 `kRepeatWarnThreshold=4`, `kSameToolWarnThreshold=8`。
- bug.txt 看到的 `sameToolStreak=8 injecting warning` 是**警告还没到 break 阈值**，所以确实又跑了几轮。

**修改项**：

| 改动 | 文件:行 | 做法 |
|---|---|---|
| 拉低同工具 break 阈值 | `QueryEngine.cpp:493` | `kSameToolBreakThreshold` 从 15 → 8；`kSameToolWarnThreshold` 从 8 → 4。本次案例 8 次同名工具早就该断。 |
| 同文件重复 Read 单独拦截 | 新加 `lastReadPaths_` 状态 + Read 工具调用时 dedupe | LLM 反复整读 `index.html` 5 次以上 → 直接返回 cache 而不是再读；或者直接拒绝并提示"already read, refer to prior result"。 |
| 同失败 shell 命令拦截 | `BashTool.h` 内部加 `lastFailedCmds_` set | 同一字符串命令连续失败 3 次后下次同命令直接拒绝，让 LLM 必须换方法。 |
| Break 后不只是 warning，应**回退到 AskUser** | `QueryEngine.cpp:1000-1004` | 当前是 `break + onText("stopping")`。改成：自动调用 AskUserQuestion 把"我卡在 X，要不要 A/B/C"丢给用户，比直接结束有用。 |

## 🟡 v1 P2 上下文瘦身 — 工具层已就位，缺一处主动 dedupe

**实测代码**：
- `AgentTool.h:54-69` 已经实现">2KB 持久化到磁盘 + preview"（JackProAi 风格）。
- `FileWriteTool.h:60-67` 和 `FileEditTool.h:173-178` 的 tool_result 已经只回一句确认，不带 diff。
- 主线程的 messages 没有 dedupe — 同一文件被 Read 5 次，5 份 4KB 全在历史里。

**修改项**：

| 改动 | 文件:行 | 做法 |
|---|---|---|
| FileReadTool 加 dedupe | `FileReadTool/FileReadTool.h` | 在 ToolContext 持一份 `recentlyReadFiles: Map<path, (hash, msgIndex)>`。同 path + 同 mtime + 同 range → tool_result 替换为 `"[same as turn N's read of <path>; content unchanged]"`。 |
| Bash 长输出统一持久化 | `BashTool.h:97-100` | 当前是输出 > 100KB 就截断。改成：> 20KB 时存磁盘 + 返回 2KB preview（复用 AgentTool 的模式）。 |
| 主线程定期 microcompact | `QueryEngine.cpp` 每轮入口 | 计算当前 messages 序列化字节数，> 80KB 时调一次"把 4 轮前的 tool_result 全部替换为 [cleared]"，**用消息替换 API 而不是 in-place mutate**（保住 cache）。 |

## 🟢 新发现 4 — 模型名建议

- 配置 `config/config.yaml:24` 还是 `claude-opus-4-7`。
- bug.txt 看到的报错是 `403 预扣费额度失败`（余额不足），不是 503（模型未知），说明**模型名当前是通过的**。但 `claude-opus-4-7` 在某些中转还是可能不认。
- 建议：保留 `claude-opus-4-7` 作为默认，但加一个 fallback 链 `claude-opus-4-7 → claude-opus-4-6 → claude-opus-4-20250514`，503 触发时按链路尝试。已有 `RemoteAPIClient` 接收 `modelOverride` 参数（line 26），加 retry 包装即可。
